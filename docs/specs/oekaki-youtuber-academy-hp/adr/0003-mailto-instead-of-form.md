# ADR-0003: 問い合わせは mailto: で代替し、HTML フォームを置かない

| 項目 | 値 |
|---|---|
| 状態 | Accepted |
| 日付 | 2026-06-28 |
| 関連要件 | FR-703, NFR-L1 |

---

## 1. 文脈・問題

ブランドホームページに「お問い合わせ」セクションが必要だが、HTML フォーム実装は静的サイトには重い（バックエンド / バリデーション / SPAM / GDPR 配慮）。一方で、訪問者には素早く問い合わせ手段を提供したい。

## 2. 検討した選択肢

| 案 | 概要 | 利点 | 欠点 |
|---|---|---|---|
| A. `<form>` + Vercel Functions | サーバーレスで処理 | 体験良好 | バックエンド管理、SPAM 対策、PII 取扱で複雑化 |
| B. `<form>` + Formspree / Netlify Forms 等の SaaS | 外部 SaaS で受信 | 楽 | 月額・無料枠制限・外部依存 |
| C. **`mailto:` + 平文表示** | リンクと表示両方 | バックエンド不要、コスト 0、PII 収集ロジック不要 | スパムボット収集リスク、スマホで起動するメーラー次第で UX 不揃い |
| D. お問い合わせなし | LINE 経由に統一 | 最小構成 | 「窓口がない」と取材側に不信感 |

## 3. 決定

**案 C: `mailto:` リンク + メールアドレスの平文表示** を採用。

### 実装ポイント

- `<a href="mailto:info@example.com?subject=...">` ボタン
- URL エンコードした subject に「【お問い合わせ】お絵かきYouTuberアカデミー」を入れる
- 平文でもメールアドレスを表示し、コピペ用にする（mailto が動かない環境でのフォールバック）
- HTML `<form>` を一切置かない（PII 収集なし → NFR-L1 が発火しない）

## 4. 結果

- 取材依頼者・受講検討者・メディア側は 30 秒以内に問い合わせ開始可能（US-003）
- バックエンドゼロ
- スパム対策は受信側のメールフィルタ（Gmail 等）に委譲
- プライバシーポリシーの法的要否を回避

## 5. トレードオフ

- メールアドレスがクローラに収集される（threat-model I-5）
  - 軽い obfuscation（`info [at] example [dot] com`）を将来検討
- アドレス変更時は html を編集 → push が必要（運用負荷は軽微）
- 受信側のメールアドレス情報セキュリティに完全依存
